# Call Me Maybe



## Description
**call me maybe** is a function-calling system that translates natural language prompts into structured JSON function calls using a small language model (Qwen3-0.6B).

Given a natural language request such as `"What is the sum of 2 and 3?"`, the system does not answer the question directly. Instead, it identifies the appropriate function to call and extracts the required arguments:

```json
{
  "prompt": "What is the sum of 2 and 3?",
  "name": "fn_add_numbers",
  "parameters": {"a": 2.0, "b": 3.0}
}
```

The core technique is **constrained decoding**: rather than prompting the model and hoping for structured output, the system intervenes at the logit level on every generation step, masking invalid tokens to `-inf` before sampling. This enforces both the selection of a valid function name and the generation of type-correct parameter values, without relying on any external grammar-enforcement library.


## Implementation
JSON structure is assembled deterministically by the program — the model never emits braces, commas, or quotes.
The model's only job is to decide **which function to call** and **what values to assign to each parameter**. Constrained decoding ensures those decisions are drawn from the valid candidate set.

### Algorithm Explanation

#### Constrained Decoding
Language models generate text one token at a time. At each step, the model computes a probability distribution (logits) over every token in the vocabulary. Normally, the highest-scoring token is selected. Constrained decoding intervenes **before** that selection by setting the logits of any token that would violate the current constraint to `-inf`, so that only valid tokens remain candidates.

In this project, constrained decoding is applied at exactly two points in the pipeline: the function name and the parameter values. The surrounding JSON structure is assembled deterministically by the program — the model never generates braces, commas, or quotation marks itself. As a result, structural validity is guaranteed by construction, while constrained decoding is used only where the model must make meaningful semantic choices.

#### Function Calling

##### 1. Function Selection
The model receives a prompt constructed from the user's request and the list of available function names with their descriptions. Logits are then masked to allow only tokens that can begin a valid function name from the provided definitions. The highest-scoring surviving token determines which function is selected.

```
User request:  "What is the sum of 2 and 3?"
Allowed set:   {"fn_add_numbers", "fn_greet", "fn_reverse_string", ...}
Masked logits: all other tokens → -inf
Result:        "fn_add_numbers"
```

##### 2. Parameter Value Selection
For each parameter declared in the selected function's schema, the model generates a value token-by-token with logits masked to a type-appropriate set:

| Parameter type | Allowed tokens                                       |
|----------------|------------------------------------------------------|
| `number`       | digits (`0–9`), `-`, `.`                             |
| `string`       | all characters                                       |
| `boolean`      | tokens forming `true` or `false`                     |

The vocabulary JSON file (obtained via `get_path_to_vocab_file()`) provides a mapping from every token string to its ID. This mapping is used at each generation step to determine which tokens can validly extend the value being built.

##### 3. Deterministic Assembly
Once all values are generated, the program assembles the final output:
```json
[
    {
        "prompt": "What is the sum of 2 and 3?",
        "name": "fn_add_numbers",
        "parameters": {"a": 2.0, "b": 3.0}
    },
    {
        "prompt": "Reverse the string 'hello'",
        "name": "fn_reverse_string",
        "parameters": {"s": "hello"}
    }
]
```
The output file is written with `json.dump()`. No model involvement occurs in any
structural token.

### Design Decisions

#### Where in the pipeline the LLM should be used
The JSON envelope is built by the program rather than generated by the model. This makes structural validity a property of the code itself, and frees the model from spending forward passes on tokens (braces, quotes, key names) that are fully determined by the schema. The LLM is invoked only at the two positions where it provides genuine value: selecting which function to call, and generating each parameter's value. Constrained decoding is applied at exactly these positions with type-appropriate vocabularies.

#### Double-quote as generation terminator instead of EOS token
String parameter generation stops when the model emits a double-quote character (") rather than an EOS token. This design was chosen for several practical reasons. First, the llm_sdk API provides no public method for retrieving the EOS token ID of the loaded model, making reliable EOS detection impossible without depending on private internals. Second, EOS tokens differ across models, so hard-coding a token ID would unnecessarily couple the implementation to Qwen3-0.6B and risk silent failure on other models. Finally, within the JSON schema used in this project, a standalone double-quote never appears inside a valid parameter value; it only appears at the boundary between the value and the surrounding JSON structure. This makes the double-quote an unambiguous, model-agnostic, and structurally reliable termination signal.

### Performance Analysis
| Metric                 | Result                           |
|------------------------|----------------------------------|
| JSON validity          | 100%                             |
| Function-call accuracy | 100% on all provided test cases  |
| Processing time        | 2m30s for all provided test cases|

Because the model only needs to select a function name from a known list and emit type-appropriate values for each parameter, the constrained token set at each step is small and well-defined. Hallucinated function names and type mismatches are ruled out at the generation level. The model's capacity is spent entirely on the semantic task rather than on producing structural tokens.

### Challenges Faced

#### Whether the LLM should classify strings as regular expressions or literal strings
I initially tried to classify string parameters as `regex` or `literal` using the LLM's first-token logits. It failed systematically: every parameter of `fn_substitute_string_with_regex` was classified as `regex` because the function name dominated the logit distribution. *Brief is Better* (arXiv:2604.02155) reports that even on Qwen2.5-1.5B (more than twice our model's size), 30.5% of function-calling tasks fail at zero-reasoning due to selection bias from surface cues. Based on this result, I ultimately chose not to perform any semantic classification of string parameters and instead treated all string values uniformly during decoding.

### Testing Strategy
1. **Provided test cases** — ran the full `function_calling_tests.json` and verified
   that every output entry matched the expected function name and parameter types.
2. **Rephrased prompts** — tested with alternative phrasings of the same requests to
   check robustness of function selection beyond the exact wording in the provided
   inputs.
3. **Malformed input handling** — tested with a missing input file and a file
   containing invalid JSON to verify graceful error handling rather than a crash.
4. **Lint** — ran `make lint` (`flake8` + `mypy`) and resolved all reported issues
   before submission.
